Creating Custom Hooks

Let's take a step back and do a quick review on what we know about React so far, which, admittedly, is quite a bit.

React is a library for building user interfaces. With React, your View a function of your application's State – v = f(s). All you have to do is worry about how the state in your application changes, and React handles the rest.

The primary mode of encapsulation for this concept is the component – which encapsulates both the visual representation of a particular piece of UI as well as the state and logic that goes along with it.

By doing so, the same intuition you have about creating and composing together functions can directly apply to creating and composing components. However, instead of composing functions together to get some value, you can compose components together to get some UI.

To accomplish this, React needed a way to allow you to describe what the UI for a component would look like from inside the component itself.

This thinking led to the creation of JSX - a wonderful lovechild of HTML and JavaScript that allows you to write HTML-ish looking syntax directly inside of JavaScript.

What made JSX different was it combined the power and expressiveness of JavaScript with the readability and accessibility of HTML. If you were already comfortable with both JavaScript and HTML (which you probably were, if you were building web apps), then you could get up to speed with JSX in a matter of minutes.

function Profile({ user }) {
  return (
    <div className="profile">
      <Avatar username={user.avatar} />
      <Bio text={user.bio} />
      <ul>
        {user.preferences.map((preference) => (
          <li>
            <Setting
              key={preference.id}
              preference={preference}
            />
          </li>
        ))}
      </ul>
    </div>
  );
}

At this point, components were a beautiful solution for encapsulating and composing UI. In fact, when you think about composition in React, odds are you think in terms of UI composition. This is natural since it's what React is so good at.

The problem is in the real world, there's more to building an app than just the UI layer. It's not uncommon to need to compose and reuse non-visual logic as well. However, because React couples UI to a component, historically, this was difficult.

This is the problem that hooks were created to solve. Just like a component enabled the composition and reusability of UI, hooks enabled the composition and reusability of non-visual logic.

  • useState
    create a value that is preserved across renders and triggers a re-render when it changes
  • useEffect
    synchronize a component with some external system
  • useRef
    create a value that is preserved across renders, but won't trigger a re-render when it changes
  • useContext
    get access to what was passed to a Context's Provider
  • useReducer
    create a value that is preserved across renders and triggers a re-render when it changes, using the reducer pattern
  • useMemo
    cache the result of a calculation between renders
  • useCallback
    cache a function between renders
  • useLayoutEffect
    synchronize a component with some external system, *before* the browser paints the screen
  • useSyncExternalStore
    subscribe to an external store
  • useEffectEvent
    encapsulate a side effect that synchronizes your component with some outside system

With all that said, and with all we've covered so far, there's still one more aspect of React that I would consider fundamental and necessary knowledge. Per usual, let's discover it with an example.

If you'll remember back to the Advanced Effects lesson, we had an app that displayed the user's current network status. We implemented it like this, using React's useSyncExternalStore hook.

import * as React from "react"

const getSnapshot = () => {
  return navigator.onLine ? "online" : "offline"
}

const subscribe = (callback) => {
  window.addEventListener("online", callback)
  window.addEventListener("offline", callback)

  return () => {
    window.removeEventListener("online", callback)
    window.removeEventListener("offline", callback)
  }
}

export default function App () {
  const networkStatus = React.useSyncExternalStore(
    subscribe, 
    getSnapshot
  )

  return (
    <div>
      <span className={networkStatus} />
      <label>{networkStatus}</label>
    </div>
  )
}

Now here's a question I want you to think through. Let's say we wanted to display the user's network status in another part of our app. How would you go about doing that?

My guess is your first intuition would be to utilize this One Cool Trick™️.

How To Reuse Non-Visual Logic

"CMD + C" then "CMD + V"

Because we've coupled the logic for determining the user's network status to the component that displays it, if we wanted to reuse that logic, we'd be forced to copy and paste it into another component.

import * as React from "react"
import Banner from './Banner'


const getSnapshot = () => {
  return navigator.onLine ? "online" : "offline"
}


const subscribe = (callback) => {
  window.addEventListener("online", callback)
  window.addEventListener("offline", callback)


  return () => {
    window.removeEventListener("online", callback)
    window.removeEventListener("offline", callback)
  }
}


export default function OfflineBanner () {
  const networkStatus = React.useSyncExternalStore(
    subscribe, 
    getSnapshot
  )


  if (networkStatus === "online") {
    return null
  }


  return (
    <Banner emoji="🛜" title="You're currently offline.">
      Please re-connect to the internet before continuing.
    </Banner>
  )
}

Isn't the whole purpose of hooks, as mentioned earlier, to enable the composition and reusability of non-visual logic? If that's the case, copy/pasting non-visual logic all over our app doesn't seem like the most elegant solution.

What if, just like React comes with its own set of hooks, we were also able to make our own custom hooks? In that scenario, a custom useNetworkStatus hook would be awfully convenient.

import * as React from "react"
import Banner from './Banner'
import useNetworkStatus from "./useNetworkStatus"


export default function OfflineBanner () {
  const networkStatus = useNetworkStatus()


  if (networkStatus === "online") {
    return null
  }


  return (
    <Banner emoji="🛜" title="You're currently offline.">
      Please re-connect to the internet before continuing.
    </Banner>
  )
}

As you probably guessed by now, not only is this possible, but it's highly encouraged whenever you want to re-use non-visual logic.

Here's what that custom useNetworkStatus hook would look like.

import * as React from "react"

const getSnapshot = () => {
  return navigator.onLine ? "online" : "offline"
}

const subscribe = (callback) => {
  window.addEventListener("online", callback)
  window.addEventListener("offline", callback)

  return () => {
    window.removeEventListener("online", callback)
    window.removeEventListener("offline", callback)
  }
}

export default function useNetworkStatus () {
  return React.useSyncExternalStore(
    subscribe, 
    getSnapshot
  )
}

Notice that useNetworkStatus is just a function that starts with use and it uses another hook (useSyncExternalStore) inside of it. That is (literally) the definition of a custom hook.

What is a Custom Hook?

A custom hook is a function that starts with use and calls other hooks inside of it

Now, anywhere inside of our app where we need the user's network status, we can use our networkStatus hook.

One question you may have is why exactly does a hook need to start with use? That seems overly restrictive.

The reason is so it's clear where the "React-y" features of the components are, whether that's state, effects, or any other hooks. If you see a function call that starts with use, you can be sure that it's utilizing other hooks inside of it. On the contrary, if you see a function call that doesn't start with use, say getTheme, you can be sure that it's just a normal JavaScript function and has nothing to do with React.

The second part of the definition is also important. If the function doesn't call other hooks, whether those hooks are built-in React hooks like useState or useEffect, or other custom hooks, then it's not a hook – it's just a function – and doesn't need to start with use.

Now the only question left is when is it appropriate to create a custom hook? The answer, as with most things in React, is to rely on your intuition. The same intuition you've developed about functions and when to create them can be directly applied to creating hooks. Mostly, you'll know it when you see it.
