Managing Effects

React, in its purest form, is a library for building user interfaces. It's so simple that the entire mental model can be represented as a formula.

A not so often talked about aspect of this formula is one simple but requisite rule in React – that the function which calculates the View needs to be a pure calculation.

That sounds fancy, but in the context of React, it just means that when React , it should be able to get a description of the UI without running into any side effects.

🤧

A Gentle Reminder

A function has a side effect any time it does anything other than take some input, an argument, and calculate some output, a return value.

Adjusted for React, a component has a side effect any time it does anything other than take some input, props and state, and calculate some output, a View.

API calls, manual DOM manipulation, using browser APIs like localStorage or setTimeout, or anything else that falls outside of simply calculating a View based on props and state is a side effect.

This makes sense. When React renders, it does so with the goal of eventually updating the UI. This entire process needs to be so as fast as possible. If it weren't, regardless of the cause, there wouldn't be much value in using React.

Rule #0

When a component renders, it should do so without running into any side effects

Now all of this sounds great – it's not terribly hard to see why React renders need to be pure calculations. The problem, as is usually the case, is reality often isn't so simple. Real applications aren't free of side effects. In fact, they're littered with them. So much so that I'd argue one of the biggest causes for bugs in any application is the mismanagement of side effects.

If React didn't have an escape hatch for managing side effects which fall outside of the normal v = fn(s) paradigm, it would be a perfect project that we'd geek out over in Computer Science lectures, not a library that powers an increasing percentage of the internet.

🫠

The React Industrial Complex

If you're already vaguely familiar with how to manage side effects in React, I ask you to gently forget everything you've learned about this topic.

The reason for that is there have been many think pieces dedicated to this topic, and almost every one of them is either, at worse, confidently incorrect, or at best, unnecessarily complex.

Even the official React docs (which are good, I might add), have separate, comprehensive posts all dedicated to this topic.

But if you can give me your pure, unadulterated brain, I'll give you a simple, easy to understand, and most importantly, correct mental model for managing side effects in React.

Let's start off by looking at an app you're already familiar with, our greetings app we first saw in the post about how React re-renders.

As a reminder, our app works by sticking all our greetings in an array, and then having a piece of state that keeps track of the index of the greeting we want to display,

We then have a button that when it's clicked, will increment the index or reset it back to 0 if we've reached the end of the array.

import * as React from "react"

function Greeting ({ name }) {
  const [index, setIndex] = React.useState(0)

  const greetings = ['Hello', "Hola", "Bonjour"]

  const handleClick = () => {
    const nextIndex = index === greetings.length - 1
      ? 0
      : index + 1
    setIndex(nextIndex)
  }

  return (
    <main>
      <h1>{greetings[index]}, {name}</h1>
      <button onClick={handleClick}>Next Greeting</button>
    </main>
  )
}

export default function App () {
  return <Greeting name="Tyler" />
}

This code is the perfect representation of the React model. It's simple, and pure, and as contrived as a Hallmark Christmas movie.

Let's dip our toes into the murky yet alluring pool of complexity by adding a side effect to our app. Specifically, let's store the user's preferred greeting in local storage using the browser's localStorage API. That way, when the user comes back to our app, even if they've closed the browser, they'll see the greeting they last selected.

First, before we can worry about getting the value from local storage, we need to figure out how to add the item to local storage. Your intuition may be to do something like this.

import * as React from "react"

function Greeting ({ name }) {
  const [index, setIndex] = React.useState(0)

  const greetings = ['Hello', "Hola", "Bonjour"]

  const handleClick = () => {
    const nextIndex = index === greetings.length - 1
      ? 0
      : index + 1
    setIndex(nextIndex)
  }

  localStorage.setItem("index", index)

  return (
    <main>
      <h1>{greetings[index]}, {name}</h1>
      <button onClick={handleClick}>Next Greeting</button>
    </main>
  )
}

export default function App () {
  return <Greeting name="Tyler" />
}

Now, whenever our app re-renders, local storage will get updated with the new index value, which we can then use in the future when establishing the initial state of the user's app.

This seems like a reasonable solution, and it might be, if it didn't violate .

Interacting with localStorage is a side effect and, regardless of how inconsequential or fast the side effect may seem, we need to keep it out of React's rendering flow.

So how do we do that? This brings us to our next rule.

Rule #1

If a side effect is triggered by an event, put that side effect in an event handler

As obvious as Rule #1 may seem, it's surprising how often it's neglected.

The entire point of an event handler is to encapsulate the logic for an event. In doing so, in React, you naturally decouple that logic from React's rendering flow. Therefore, if a side effect is triggered by an event, put that side effect where you put all the other logic for that event, in its event handler.

In the context of our app, that means moving our side effect, localStorage.setItem, into the event handler that triggers the event, handleClick.

import * as React from "react"

function Greeting ({ name }) {
  const [index, setIndex] = React.useState(0)

  const greetings = ['Hello', "Hola", "Bonjour"]

  const handleClick = () => {
    const nextIndex = index === greetings.length - 1
      ? 0
      : index + 1
    setIndex(nextIndex)

    localStorage.setItem("index", nextIndex)
  }

  return (
    <main>
      <h1>{greetings[index]}, {name}</h1>
      <button onClick={handleClick}>Next Greeting</button>
    </main>
  )
}

export default function App () {
  return <Greeting name="Tyler" />
}

Notice our side effect is still part of our component, we've just abstracted it to a part of our component that's . React is still aware of handleClick, but what's inside of it, side effects or not, are irrelevant during render.

Now that we've cached our index in local storage, we need to figure out how to get it back out when we set the initial index of our app.

Your first intuition might be to do something like this.

import * as React from "react"

function Greeting ({ name }) {
  const [index, setIndex] = React.useState(
    Number(localStorage.getItem("index"))
  )

  const greetings = ['Hello', "Hola", "Bonjour"]

  const handleClick = () => {
    const nextIndex = index === greetings.length - 1
      ? 0
      : index + 1
    setIndex(nextIndex)

    localStorage.setItem("index", nextIndex)
  }

  return (
    <main>
      <h1>{greetings[index]}, {name}</h1>
      <button onClick={handleClick}>Next Greeting</button>
    </main>
  )
}

export default function App () {
  return <Greeting name="Tyler" />
}

This seems lik a reasonable solution, but again, it violates . Not only that, but it technically violates Rule #0 on every render.

Every time Greeting renders, our Number(localStorage.getItem("index")) side effect is going to be invoked, even though React will only use the result of that invocation on the initial render.

Now you may be familiar with lazy state initialization. The idea is you can pass useState a function that will only be called once, on the initial render, and the return value of that function will be used as the initial value for that state.

Something like this.

import * as React from "react"

function Greeting ({ name }) {
  const [index, setIndex] = React.useState(() => {
    return Number(localStorage.getItem("index"))
  })

  const greetings = ['Hello', "Hola", "Bonjour"]

  const handleClick = () => {
    const nextIndex = index === greetings.length - 1
      ? 0
      : index + 1
    setIndex(nextIndex)

    localStorage.setItem("index", nextIndex)
  }

  return (
    <main>
      <h1>{greetings[index]}, {name}</h1>
      <button onClick={handleClick}>Next Greeting</button>
    </main>
  )
}

export default function App () {
  return <Greeting name="Tyler" />
}

This is better, as our side effect will only get called once on the initial render. Unfortunately, even if it's just once, we're still violating .

Now I know what you might be thinking, surely this isn't that big of deal, right? Getting a single item out of local storage is plenty fast. Even if it slightly slows down the initial render, does it really matter?

Rebellious of you, but we have rules for a reason. In this case, if you tried to server render our app (which is a totally reasonable thing you could do), you'd get an error since localStorage isn't available on the server, obviously.

If you adjusted your code to account for that, you'd end up with something like this. 

const [index, setIndex] = React.useState(() => {
  if (typeof window === "undefined") {
    return 0
  }


  return Number(localStorage.getItem("index"))
})

Which seems reasonable, but if there was a value in local storage on the initial render on the client, React would give you another error

failed because the initial UI does not match what was rendered on the server.

Now there are some we could get around this, or we could just follow our rules and avoid the problem entirely.

With that said, we still don't really have a rule for this scenario. We know what not to do by following , but that doesn't get us anywhere closer to knowing where to put our side effect.

This brings us to our next rule.

Rule #2

If a side effect is synchronizing your component with some external system, put that side effect inside useEffect

useEffect is a hook that comes with React that, as our definition suggests, allows you to run a side effect that synchronizes your component with some outside system. useEffect works by removing the side effect from React's rendering flow and delaying its execution until after the component has rendered.

Conceptually this makes sense, especially when combined with . React can maximize the speed and predictability of rendering by enforcing rules around when side effects can run – either when an event occurs (Rule #1) or after the component has rendered (Rule #2).

Let's look at another example.

👹

Strict Mode

Note that all of our examples have "Strict Mode" disabled. As a reminder, Strict Mode is a way for React to stress test your components in development.

Initially, I'm more interested in helping you get a working mental model for how React works in production. We'll talk about how Strict Mode affects effects later in this post – but as always, you can toggle Strict Mode on if you'd like to play with it yourself.

As a gentle example of useEffect, here's a component that show's the user's current battery level. We'll use useEffect since we want to synchronize the component's state with some outside system (the battery level on the user's device).

I've thrown in some logs so you can see the order of things.

import * as React from "react"

export default function BatteryLevel() {
  const [level, setLevel] = React.useState(0)

  React.useEffect(() => {
    console.log("Getting battery level...")
    navigator.getBattery().then(battery => {
      const newLevel = Math.round(battery.level * 100)

      if (newLevel !== level) {
        setLevel(newLevel)
      }
    })
  })

  console.log("Rendering")
  return (
    <p>{level}%</p>
  )
}

Even though it's basic, there are a few interesting things to note about this example.

First, as mentioned earlier, our gets invoked after React renders. You can see this by the fact that Rendering gets logged before Getting battery level....

Second, and this may come as a surprise to you, by default, our effect gets invoked after every re-render. This is why the console output looks the way it does.

It's a little long winded, but it makes sense when you break down all the steps.

On the initial render we set the level state is to 0 and we log Rendering. Then, after rendering, React calls our effect which logs Getting battery level... before asynchronously fetching our device's battery level. Once it has it, if the battery level is different than our initial state of 0 (which is a safe assumption), React re-renders with the new battery level.

On the re-render, Rendering gets logged and then, after React has finished rendering, our effect is called. The effect logs Getting battery level... before asynchronously fetching the battery level, again. Since this time the battery level is the same as the previous render, our effect ends and React is done.

🧾

Prove it

If you wanted to test our logic, go up to the code above and change the initial level state to whatever your current battery level is and refresh the preview.

What you'll see is that, because the battery level is the same as the initial state, React won't re-render and the effect is only called once.

Admittedly, the default behavior of React calling the effect after every render might seem a little strange. In both of our examples, that's not ideal.

Instead, really all we want to do is run some side effect on the initial render, set that value to state, and then be done with it. Continuing to call the side effect on every render is a waste. Thankfully, useEffect accepts a second argument that gives us a little bit more control over when our effect is called.

Now before we dive into it, I want to warn you that it probably doesn't work how you're expecting it to. This is such a common misconception, I want to call it out explicitly.

Because I presume you have a normal human brain, what you're probably expecting is something like this, where our second argument to useEffect tells React when to invoke the effect.

React.useEffect(() => {
  document.title = `Welcome, ${name}`
}, ["onInitialRender"])

This is how everyone about the second argument, but it's wrong. Instead of telling React when to invoke the effect, you give React all the dependencies that effect needs to run – like this.

React.useEffect(() => {
  document.title = `Welcome, ${name}`
}, [name])

Now, whenever , React will re-run the effect.

It may seem strange, but this default makes sense if you think about it in the context of the goal of useEffect. The whole goal of useEffect is to synchronize your component with some external system. Whenever any of the dependencies that the effect needs in order to synchronize change, React, should resynchronize.

Now with that said, how can we use this knowledge of the dependency array to update both of our examples?

If you look back at both examples, neither actually depend on any values outside of the effect – meaning, neither have any dependencies.

So...let's tell React that by giving React an empty array as our second argument.

First, our battery example.

import * as React from "react"

export default function BatteryLevel() {
  const [level, setLevel] = React.useState(0)

  React.useEffect(() => {
    console.log("Getting battery level...")
    navigator.getBattery().then(battery => {
      const newLevel = Math.round(battery.level * 100)

      if (newLevel !== level) {
        setLevel(newLevel)
      }
    })
  }, [])

  console.log("Rendering")
  return (
    <p>{level}%</p>
  )
}

Notice that our effect only gets invoked once, after the initial render. This makes sense. We've told React that the effect doesn't depend on any values outside of the effect, so it never needs to re-run (or resync).

And for our local storage example, we can do the same thing.

import * as React from "react"

function Greeting ({ name }) {
  const [index, setIndex] = React.useState(0)

  const greetings = ['Hello', "Hola", "Bonjour"]

  const handleClick = () => {
    const nextIndex = index === greetings.length - 1
      ? 0
      : index + 1
    setIndex(nextIndex)

    localStorage.setItem("index", nextIndex)
  }

  React.useEffect(() => {
    const item = localStorage.getItem("index")

    if (item) {
      setIndex(Number(item))
    }
  }, [])

  return (
    <main>
      <h1>{greetings[index]}, {name}</h1>
      <button onClick={handleClick}>Next Greeting</button>
    </main>
  )
}

export default function App () {
  return <Greeting name="Tyler" />
}

This example is now the perfect representation of our 3 rules. We've successfully moved all of our side effects out of React's by moving setItem to an and getItem to .

And most important, you now have a first principles understanding for how to manage side effects in React, and you can let those principles guide you in your decision making for all effects you encounter, not just the ones you've seen before.

Keep this in mind as we move on to more advanced scenarios in the next lesson. If you stick to our rules, no matter how difficult the effects seem, you'll be fine.

Rule #0

When a component renders, it should do so without running into any side effects

Rule #1

If a side effect is triggered by an event, put that side effect in an event handler

Rule #2

If a side effect is synchronizing your component with some outside system, put that side effect inside useEffect
